## P1 — wrong-but-not-broken

### P1-1. `entities.ts` — 14 mutations lack owner/partner auth check
**Files:** `src/app/actions/entities.ts` (lines 60, 80, 102, 114, 142, 161, 218, 244, 282, 348, 397, 489, 498, 532)

`createClientAction`, `updateClientAction`, `createBrandAction`, `createTalentAction`, `updateTalentAction`, `createCrewAction`, `updateCrewAction`, `deleteCrewAction`, `deleteTalentAction`, `anonymiseTalentAction`, `anonymiseClientAction`, `anonymiseCrewAction`, `setTalentActiveAction`, `setCrewActiveAction` all skip the `getCurrentAppUser().role ∈ {owner, partner}` guard that `bookings.ts` enforces.

These routes are behind `(dashboard)/` which requires auth, so an unauthenticated request can't reach them. But a talent or crew portal user with a session cookie could in theory call them from devtools. Defence-in-depth gap.

**Fix:** Add `const user = await getCurrentAppUser(); if (!user || (user.role !== 'owner' && user.role !== 'partner')) return { error: 'Forbidden' };` to each.

### P1-2. `locations.ts` — 3 actions, 0 auth
**File:** `src/app/actions/locations.ts:48, 73, 88`

`createLocationAction`, `updateLocationAction`, `retryLocationDriveFolderAction` have no caller-identity check, no audit log, and no permission gate. Same risk as P1-1 but smaller blast radius (locations are reference data).

**Fix:** Same pattern as P1-1; also wrap mutations in `auditEntityMutation` equivalent.

### P1-3. Portal mutations skip booking-cache busting
**File:** `src/app/actions/portal.ts:74, 123, 160, 196, 220, 249, 273, 301`

All 8 portal actions (`respondToCrewHoldAction`, `respondToTalentHoldAction`, `acceptTalentRateAction`, `acknowledgeBriefAction`, `addUnavailabilityAction`, `removeUnavailabilityAction`, plus crew equivalents) call `revalidatePath('/portal/...')` only. None calls `revalidateTag('bookings', {})`.

**Impact:** When Mason confirms a hold from `/portal/crew`, the owner's view of `/bookings/[id]` shows him still pencilled until the dashboard cache TTL (60s) elapses. Same for talent rate acceptance and brief acknowledgement. The data IS persisted — only the owner's UI lags.

**Fix:** Add `revalidateTag('bookings', {})` after each `revalidatePath` in portal.ts.

### P1-4. `HoldExpiryBadge` hardcodes Sydney +10:00 — 1h off during AEDT
**File:** `src/components/bookings/HoldExpiryBadge.tsx:83`

```ts
const iso = new Date(`${draft}T09:00:00+10:00`).toISOString();
```

NSW observes daylight savings Oct→Apr. During AEDT the correct offset is +11:00. A hold "expires Tuesday" set during summer actually saves 1h earlier in UTC than intended. The day-precision badge masks the issue most of the time but the underlying timestamp is wrong.

**Fix:** Compute the offset dynamically via `Intl.DateTimeFormat` with `timeZone: 'Australia/Sydney'`, or store just the date (day precision) and let the renderer apply local-time end-of-day on read.

### P1-5. `SchedulesPanel` Delete has no confirm
**File:** `src/components/bookings/SchedulesPanel.tsx:85–87, 159`

```tsx
function handleDelete(s: BookingSchedule) {
  startTransition(async () => {
    const result = await deleteScheduleAction(s.id, bookingId);
```

Single click on the bin icon permanently deletes a per-day schedule row. Compare to `QuoteBuilder.tsx:232` which uses `confirm('Remove this fee line?')`.

**Fix:** Wrap `handleDelete` in `if (!confirm('Delete schedule for {date}?')) return;` or add the same two-step reveal pattern used by `DeleteEntityButton`.

### P1-6. Hard-delete confirmation UX inconsistent across the app
**Files:** `src/components/entities/DeleteEntityButton.tsx` vs `src/components/entities/DataRightsControls.tsx` vs `src/components/bookings/BookingLifecycleControls.tsx`

| Destructive action | Confirmation pattern |
|---|---|
| Booking Delete | Type "DELETE" to enable button |
| Talent/Client Anonymise | Type "ANONYMISE" to enable button |
| Crew Anonymise | Type "ANONYMISE" to enable button |
| Talent/Crew Hard Delete | 2-step click reveal |
| Schedule Delete | No confirm (P1-5) |
| Fee line Delete | `confirm()` dialog |

Three different UX patterns for similar gravity. Talent/Crew hard delete carries the same data-loss risk as booking delete but uses a weaker confirm.

**Fix:** Standardise on type-to-confirm for all hard deletes and anonymisations. Use a shared `<TypeToConfirmButton word="DELETE">` component.

### P1-7. Crew detail missing Archive button
**Files:** `src/app/(dashboard)/crew/[id]/page.tsx` (action row) vs `src/app/(dashboard)/talent/[id]/page.tsx:142`

Talent detail header has `<ArchiveTalentButton talentId={...} currentlyActive={...} />`. Crew detail has only `<DeleteEntityButton>`. Soft-archive via `setCrewActiveAction` exists in `entities.ts:498` but no UI surfaces it.

**Fix:** Add `<ArchiveCrewButton>` mirroring `ArchiveTalentButton` and wire it into the crew detail action row.

### P1-8. Client detail header inconsistent with talent/crew
**File:** `src/app/(dashboard)/clients/[id]/page.tsx:104–135`

Talent + crew detail headers use a 3-row structure: identity / status badges / actions. Client detail uses a single row with action buttons inline. No badges row, no Archive, no Delete. Just Drive ↗ and Edit.

**Fix:** Rebuild client detail header to match the talent/crew 3-row pattern. Add an `ArchiveClientButton` (using `is_archived` if present, or `is_active`) and consider whether clients should expose `DeleteEntityButton` like talent/crew do.

### P1-9. `BookingDetail` client link falls back to dead `href="#"`
**File:** `src/components/bookings/BookingDetail.tsx:96`

```tsx
href={booking.client?.id ? `/clients/${booking.client.id}` : '#'}
```

When a booking has no client (rare but possible — `client_id` is nullable in the schema), the chip renders as a clickable anchor that scrolls to top of page. Should render as plain text instead.

**Fix:** Conditional: render `<Link>` when `client?.id` truthy, else render a `<span>` with the same styling.

### P1-10. `quotes-public.ts` accept/decline lacks explicit audit row
**File:** `src/app/actions/quotes-public.ts:17–98`

`acceptQuoteByTokenAction` and `declineQuoteByTokenAction` call `transitionState()` which writes an audit row for the state change, but there's no dedicated `quote_accept_by_token` / `quote_decline_by_token` audit row capturing that the action came from the public link (vs an owner-side approval). Useful for forensics if a client disputes.

**Fix:** Add `await logAudit({ action: 'quote_accept_by_token', tableName: 'atelier_bookings', recordId: booking.id, newValue: { token, userAgent: ... } })`.

### P1-11. `ApprovalQueue` page warning persists across filter tabs
**File:** `src/components/approvals/ApprovalQueue.tsx:32–35` + `src/app/(dashboard)/inbox/page.tsx`

The filter tabs at the top of `/inbox` are Next.js `<Link>` components changing query params. The `ApprovalQueue` component's local `pageWarning` state survives the navigation because React doesn't unmount the component when only query string changes.

**Fix:** Pass `key={filter}` to `<ApprovalQueue>` in the parent page so state resets on tab change.

## P2 — polish

### P2-1. Hardcoded hex codes outside the palette (29 instances, excluding Grid Planner)
**Files:**
- `src/components/quotes/QuoteActions.tsx` — 17 hex codes for the public quote viewer (greens, reds, oranges). Intentional white-background context, but should use light-mode CSS vars.
- `src/components/approvals/ApprovalQueue.tsx:154` — `#6b6b6b` should be `PALETTE.muted`.
- `src/components/bookings/BookingsCalendar.tsx:279` — `#404560` fallback could use `PALETTE.muted`.
- `src/components/costs/SeedButton.tsx:23` — `#262626` / `#8b8b8b` from old dark theme.
- `src/components/quotes/QuoteBuilder.tsx:1096, 1123, 1153` — `accent-[#C4A882]` Tailwind class hardcodes the sand colour for native checkbox accent. Acceptable as fallback but could move to a CSS variable.

### P2-2. 8+ headings bypass `.section-title` class
**Files:** `MorningAfterChecklist.tsx:115`, `BookingAdvanceButtons.tsx:124`, `HoldRequestsTrigger.tsx:38`, `SendQuotePanel.tsx:197, 292`, `BookingDetail.tsx:305`, `OTExpenseEntry.tsx:114, 124`

Inline `text-xs font-semibold uppercase tracking-wide` styling instead of the `.section-title` class introduced in PR #80. Visual result is similar but inconsistent and harder to retheme.

**Fix:** Replace inline classes with `<h3 className="section-title">`.

### P2-3. `BookingTabs` active tab not in URL
**File:** `src/components/bookings/BookingTabs.tsx:27`

`useState<TabKey>('overview')` — refreshing or sharing the URL loses the active tab. Recent agentic features (Approvals, Activity) benefit from deep-linkable URLs.

**Fix:** Sync to `?tab=...` query param via `useSearchParams` / `router.replace`.

### P2-4. `HoldExpiryBadge` default 14d reads as muted/grey
**File:** `src/components/bookings/HoldExpiryBadge.tsx:47–48`

Thresholds: `> 7d = muted, 4–7d = amber, ≤ 3d = red`. A freshly-added hold (14 days out) shows a grey badge that visually reads as "neutral / inactive". User might prefer "green / calm" for >7d to indicate "this is healthy".

**Fix:** Replace muted threshold with `PALETTE.success` (or `var(--p-success)`) for >7d.

### P2-5. `SendQuotePanel` hides Send button when blockers present
**File:** `src/components/bookings/SendQuotePanel.tsx:240–260`

When pre-flight detects red blockers, the Send button is removed from the DOM entirely instead of staying visible-but-disabled with a tooltip listing what's missing. User sees a Cancel button next to the gate list with no obvious next action.

**Fix:** Keep the Send button rendered but `disabled` when `hasBlockers`; add a tooltip naming the blocker.

### P2-6. `PortalDataRights` download link is bare `<a>`
**File:** `src/components/portal/PortalDataRights.tsx:48–59`

If the JSON export endpoint fails (5xx), the browser shows a default error page. No client-side state, no retry, no loading indicator.

**Fix:** Wrap in a client component that fetches the URL, surfaces a loading state, and presents a Save dialog on success.

### P2-7. `HoldCard`/`UnavailabilityManager` errors dismissable but no retry
**Files:** `src/components/portal/HoldCard.tsx:77–89`, `src/components/portal/UnavailabilityManager.tsx:~89`

Failed save → error appears → user can dismiss but has to find a fresh path to retry. Should expose a Retry button alongside Dismiss, or auto-clear on next successful action.

### P2-8. `CopyTeamButton` redundant cursor styling
**File:** `src/components/bookings/CopyTeamButton.tsx:90–97`

`disabled={disabled}` + `cursor: disabled ? 'not-allowed' : 'pointer'`. The `disabled` attribute already produces `cursor: not-allowed` via the user-agent stylesheet. Redundant.

### P2-9. `QuickCompose` opacity-only disabled visual
**File:** `src/components/bookings/QuickCompose.tsx:119–125`

`disabled` attribute is wired correctly, opacity drops to 0.5 as visual. Works, but opacity-only doesn't read clearly in dark themes (the button still feels live). Add `cursor: not-allowed` or a tooltip explaining why.

### P2-10. `BookingAdvanceButtons` modal accepts Enter without field validation
**File:** `src/components/bookings/BookingAdvanceButtons.tsx:144, 161, 180`

Enter key fires `confirmModal()` which checks `pendingTransition` but not the modal-specific field constraints (e.g. release reason required, fee acknowledgment required). Server will catch but UI should pre-validate.

### P2-11. `QuoteBuilder` drag-to-reorder has no tooltip/affordance
**File:** `src/components/quotes/QuoteBuilder.tsx:618`

Drag handle (`⠿`) renders at 35% opacity with no hover state and no tooltip explaining the gesture. Users likely don't discover it.

**Fix:** Add a tooltip `title="Drag to reorder"` and increase opacity on row hover.

### P2-12. `JobPnLPanel` returns null with no empty-state hint
**File:** `src/components/bookings/JobPnLPanel.tsx:75`

`if (!latestQuote || feeLines.length === 0) return null;` — the entire P&L section vanishes on new bookings, with no breadcrumb explaining why or pointing users to "create a quote first".

**Fix:** Render an empty-state card: "Job P&L appears once a quote exists for this booking."

### P2-13. `JobPnLPanel` "Paid out" shows even when zero
**File:** `src/components/bookings/JobPnLPanel.tsx:128`

When no fee lines have crew_id assignments, `paidOut.total === 0` still renders the row with `$0.00`. Reads as "we paid out zero" when really it means "no payment data yet."

**Fix:** Hide the row when `paidOut.total === 0`, or label it "—".
