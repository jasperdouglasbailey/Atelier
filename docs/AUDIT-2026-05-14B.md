## Phase 3 — Integration Health

### Gmail — P1 ACTIVE FAILURES

The audit_log contains **5 failed email operations** across the past 8 days:

| Date | Action | Booking |
|------|--------|---------|
| 2026-05-13 | `client_quote_chase_email_send_failed` | BOOK-0006 |
| 2026-05-12 | `client_quote_chase_email_send_failed` | different booking |
| 2026-05-08 | `send_quote_email_failed` | BOOK-0005 |
| 2026-05-08 | `draft_quote_email_failed` | BOOK-0005 |
| 2026-05-06 | `send_onboarding_link_failed` | talent record |

**What this means in plain English:**
- Two clients did not receive their automated quote follow-up emails (May 12–13). If their quotes go stale, this is lost revenue.
- One client did not receive their quote email at all on May 8 (both send and draft failed — two attempts, both failed).
- One talent/crew member did not receive their onboarding link email on May 6. They have no way to submit their details.

**Likely cause:** Gmail OAuth refresh token expired or was revoked. This is the single most common failure mode for Google integrations. The token granted during OAuth setup has a limited lifetime; when it expires, all Gmail operations fail silently (they log to audit_log but show no banner in the UI).

**Fix:** Navigate to Settings → reconnect Google OAuth. Check `GOOGLE_REFRESH_TOKEN` env var is current. Consider adding a visible warning banner when Gmail send fails (currently only written to audit_log — Jasper has no way to know without querying the DB).

### Cron Jobs — P1 NO AUDIT LOG ENTRIES

The audit_log contains **zero `cron_*` action entries**. Expected: a `cron_quote_chase_run`, `cron_compliance_pings_run`, etc. for each cron execution.

**Interpretation:** The 8 crons (auto-anonymise, compliance-pings, data-retention, lock-ot-windows, post-shoot-chase, quote-chase, talent-gallery-ping, tomorrow-digest) either:
1. Are not running at all (scheduler not configured in the deployment environment), OR
2. Run but don't write a success log entry (only failures get logged per `logAudit`)

Evidence that quote-chase IS running: the `client_quote_chase_email_send_failed` entries prove the cron queued approvals that Jasper approved, and then the email send step failed. So the cron itself ran — it just doesn't write a "I ran" entry on success.

**Gap:** There is no admin UI or log entry that confirms which crons ran when, what they found, or whether they're healthy. Jasper has no visibility into cron activity without querying the database directly. This is a P1 operational gap.

**Fix (immediate):** Have each cron write a `cron_<name>_run` audit entry on every execution with `{ bookingsFound: N, actionsQueued: M }` metadata. Add a "Cron Health" panel to `/settings` showing last-run timestamp per cron.

### Client Accepts Quote — P1

The client opens `/q/[token]` and clicks Accept: **1 click**.

But: `acceptQuoteByTokenAction` in `quotes-public.ts` does not call `revalidatePath` or `revalidateTag` (it's a public, unauthenticated action and can't access the server cache). The client-side `QuoteActions` component updates its own UI state to "Accepted" but does **not** call `router.refresh()`.

**Result:** Jasper's booking detail page still shows "quote_sent" badge until they manually reload. All the downstream side effects (Drive folder creation, Calendar invite, email draft) fire on the server but Jasper sees no evidence of them without reloading.

**Fix:** Add `router.refresh()` to `QuoteActions` after successful accept, in `src/components/quotes/QuoteActions.tsx`.

### Hold Requests — P1 (batch approve gap)

After `quote_confirmed`, hold requests for preferred crew are proposed and appear in the inbox. Jasper must **approve each one individually**. For a typical shoot with 4–6 crew, that's 4–6 separate inbox clicks. There is no "Approve all holds for BOOK-XXXX" action.

**Fix:** Add a batch approve button on the booking detail page for all pending hold requests attached to that booking.

### Post-Shoot → Invoice — P1

Steps: `post_production` → `final_delivery` → `invoice_issued` (all manual state transitions). Then:

There is **no `sendInvoiceAction`** in `src/app/actions/`. The invoice workflow is:
1. Click "Print Invoice" or "Invoice PDF ↓" in the booking tools row
2. Download the PDF
3. Manually attach it to an email and send from Gmail

This means every invoice sent is completely outside the system. The audit_log has no record of any invoice being sent. Xero (blocked on credentials) would fix this, but a stopgap `sendInvoiceEmailAction` using the existing Gmail integration would at least create an audit trail.

## Phase 5 — Unknown Unknowns

### P1 | Grid Planner has no persistence

Navigate to `/grid-planner`, upload images, add captions, refresh the page: **all work is lost**. `GridPlanner.tsx` is entirely client-side state (`useState`). There is no server action, no DB table, no `localStorage` save.

Jasper may be using this as a real content planning tool without knowing that a browser refresh destroys all work.

**Fix options (pick one):**
- Add `localStorage` persistence (10 lines of code — saves/loads slot data on mount/change)
- Add a prominent banner: "Temporary workspace — content is not saved. Refresh to start fresh."

### P1 | Gmail failures are invisible to Jasper

5 failed email operations in the past 8 days. None of them produce a visible banner, toast, or inbox alert. Jasper only finds out by querying the database or noticing that a client or talent didn't receive something.

**Fix:** When any email action fails and writes a `*_failed` audit entry, also create an inbox item or send an internal notification: "Email to [client] failed — check Gmail connection."

### P1 | Duplicate audit log action names

Fee line mutations write **two different action names** for the same operation:
- `fee_line_add` (15 entries) AND `add_fee_line` (54 entries) — same mutation, two names
- `fee_line_remove` (14) AND `remove_fee_line` (18) — same mutation, two names

The audit log is polluted and any analytics or search for fee line activity will miss ~25% of rows depending on which name is searched. This happened because the action name was renamed in a PR but the old name wasn't fully removed.

**Fix:** Standardise on one name. Grep for both names in `src/app/actions/` and remove the old form.

### P1 | BOOK-0006 (Peter Alexander) has 2 field update failures

`update_booking_field_failed` was logged twice on 2026-05-13 for booking BOOK-0006. The booking is currently in `artists_crew_held` state. The field updates that failed are unknown (the audit log doesn't capture which field). Jasper may have tried to update something that silently didn't save.

**Immediate action:** Check BOOK-0006 manually — verify all fields look correct, nothing is stale.

### P1 | No notification when talent/crew complete onboarding

When talent or crew submit `/onboard/[token]`, `onboarding_completed` flips to `true`. There is no approval item queued, no email to Jasper, no inbox entry. Jasper has to check `/settings/compliance` to notice the badge changed.

**Risk:** Jasper may activate talent (set `is_active = true`) before reviewing what they submitted, or may miss that they've completed onboarding entirely.

**Fix:** `submitOnboardingAction` should queue a low-priority inbox item: "Oliver Begg has completed onboarding — review and activate."

### P1 | No notification when crew respond to holds in portal

When crew accept or decline hold requests in `/portal/crew`, there is no approval, inbox item, or email to Jasper. Jasper only finds out by opening the booking detail and checking BookingTeam.

**Fix:** Hold response actions should write an approval or inbox notification: "Mason Williams accepted hold for BOOK-0006."

### P2 | Quote expiry not shown to client

Active bookings have `quote_token_expires_at` set (all current ones expire in 166–178 days, well within the 180-day window). But `/q/[token]` does not display the expiry date. If a client saves the link and opens it after expiry, they get an error with no explanation.

**Fix:** Add "This quote is valid until [date]" to the footer of `/q/[token]`. Add "Quote link expires in X days" badge on the booking detail for quotes in `quote_sent` state.

### P2 | No remittance advice to talent/crew when marked paid

`markTalentPaidAction` and `markCrewPaidAction` stamp `artist_paid_at`. No email is sent to the talent/crew. They find out when money arrives in their account. There is no `remittance_advice` action type in `approval-effects.ts`.

**Fix:** When marking paid, send a simple email: "Your fee of $X for BOOK-0006 has been processed."

### P2 | Grid planner Instagram API silently fails

`GridPlanner.tsx:281` has `.catch(() => {/* silently fall back to — placeholders */})`. If the Instagram stats API fails, the follower/post counts show as dashes with no explanation. Jasper has no way to know if the live stats are real or a silent fallback.

## Audit A — UX & Design Review

### P1 | Approval queue shows action_type only — no email preview

`src/app/(dashboard)/inbox/` renders each approval with its `action_type` (e.g. `client_quote_chase_email`). Jasper cannot see who the email goes to, what the subject is, or any body preview without opening the linked booking. This means every approval requires a context-switch to decide whether to approve.

**Fix:** The `atelier_approvals` table has a `payload` JSONB column. Add a one-line summary under the action type: "To: client@aje.com.au · Subject: Following up on your quote — BOOK-0005".

### P1 | Dashboard attention items not sorted by urgency

`ATTENTION_CONFIG` in the dashboard defines urgency levels per state, but `getAttentionItems()` returns items in DB insertion order. A `morning_after_check` booking (high urgency, OT window closing) can appear below a `brief_received` item (low urgency).

**Fix:** Sort `attentionItems` by `ATTENTION_CONFIG[item.state].urgency` descending before rendering.

### P1 | Optional vs pending checklist items look identical

In `StageChecklist.tsx`, `optional` items use `PALETTE.warning` (amber) — the same colour as `pending` (incomplete) items. Optional things read as urgent. A user scanning the checklist can't tell "I must do this" from "I can skip this."

**Fix:** Change `optional` status to `PALETTE.muted` with a `–` symbol. Reserve amber/warning for genuinely incomplete items.

### P2 | Quote builder totals panel: ASF is unexplained

The grand total breakdown shows "ASF" with no tooltip or legend explaining it means "Agency Service Fee." A new partner or client reading a printed quote wouldn't know what it is.

**Fix:** Add an `ⓘ` tooltip next to "ASF" reading "Agency Service Fee — 15% of subtotal." Already explained on individual line rows; just needs the same treatment on the totals panel.

### P2 | Schedule panel: no label explaining override relationship

`BookingFormFields` has `shoot_location`, `call_time`, `wrap_time` at booking level. `SchedulesPanel` has per-date overrides of the same fields. The call sheet reads the per-date schedule when it exists, falling back to the booking-level default. There is no label in the UI explaining this.

**Fix:** Add a muted note to SchedulesPanel: "Per-day times override the booking-level defaults used in the call sheet."

### P2 | Empty state in BookingTeam too vague

"No talent assigned yet." — doesn't guide the next action.

**Fix:** "No talent assigned yet — use the '+ Add talent' button above."

### P2 | Hold expiry badge is a server-rendered snapshot

`HoldExpiryBadge` computes "expires in X days" at page render time. If the booking detail page is kept open for several hours, the badge becomes stale. A hold that expires overnight will still show "expires in 1 day" the next morning.

**Fix:** Make `HoldExpiryBadge` a client component that re-computes every 60 seconds using `setInterval`.

## Audit B — Event Chain Verification

### State transition side-effect map

| Transition | Side effects | Silent failures? | UI refreshes? |
|------------|-------------|-----------------|---------------|
| `quote_confirmed` (via client accept) | Drive folder, Calendar event, confirmation email draft, agency notification email | ✅ Yes — all three async effects fail silently | ❌ **No** — Jasper's page does not refresh |
| `quote_confirmed` → `pre_production` | None | — | ✅ Yes |
| `shoot_live` → `morning_after_check` | Sets `ot_expenses_window_end = now + 7d` | None | ✅ Yes |
| `morning_after_check` → `post_production` | None | — | ✅ Yes |
| `post_production` → `final_delivery` | Sets `ot_expenses_locked = true`, `final_delivery_at = now`, creates Drive shared link | Drive link creation is async, silent fail | ✅ Yes (on success) |
| `final_delivery` → `invoice_issued` | Sets `invoice_issued_at = now` | None | ✅ Yes |
| `invoice_issued` → `paid` | Sets `paid_at = now`, refreshes client avg DOI | DOI refresh is async | ✅ Yes |

### P1 | Three async effects fail silently on quote_confirmed

When the booking transitions to `quote_confirmed` (triggered by client accepting on `/q/[token]`):
- **Drive folders** create asynchronously — if creation fails, `drive_root_id` stays null, Workspace link doesn't appear, no error shown
- **Google Calendar invite** creates asynchronously — if creation fails, team gets no invite, no error shown
- **Confirmation email draft** creates asynchronously — if Gmail is failing (which it is, per audit_log), the draft is never created, no error shown

All three use `reportDataError` which logs but doesn't surface to the UI.

**Fix:** After each async side effect completes (or fails), write to a `sideEffectStatus` field or display a non-blocking status row on the booking detail: "✅ Drive folder created · ❌ Calendar invite failed — check Google connection."

### P1 | `acceptQuoteByTokenAction` doesn't trigger Jasper's page refresh

The public accept action (`src/app/actions/quotes-public.ts`) cannot call server-side revalidation because it runs in a public context. The `QuoteActions` client component sets its own UI to "Accepted" but does not call `router.refresh()`.

**Fix:** Add `router.refresh()` in `QuoteActions.tsx` after the accept action returns `ok: true`. One line.

### P1 | `proposeHoldRequests` is fire-and-forget

When transitioning to `artists_crew_held`, `proposeHoldRequests(id)` fires without `await`. If it fails, no inbox items appear and Jasper doesn't know. Currently the only detection is checking the inbox is empty.

**Fix:** Await the call and surface failure as a booking-level warning banner.

## Audit C — Data Propagation

### P1 | Booking title not regenerated when talent removed

`atelier_bookings.title` (format: `BOOK-0042 - Oliver Begg - AJE, Resort 26`) is set at creation and not updated when talent is added or removed from `booking_talent`. After removing Oliver from the booking, the title in the list and calendar still reads his name.

**Fix:** `removeBookingTalentAction` should regenerate the booking title from remaining talent + client + brand and call `updateBooking({ title: newTitle })`.

### P1 | SchedulesPanel doesn't auto-populate when shoot dates change

When shoot dates are edited on the booking edit form, `updateBookingAction` fires but does NOT auto-create/remove schedule rows for the new dates. Jasper must manually add per-day schedule entries that match the new dates. The call sheet will have gaps for any new days with no schedule row.

**Fix:** After `updateBookingAction` saves new `shoot_dates`, generate a schedule row for each new date that doesn't already have one (upsert with conflict-ignore).

### P1 | Hold expiry badge stale after page load (see Audit A)

Same issue as above — server-rendered snapshot, not live.

### P2 | Shoot dates change → Calendar event description may be incomplete

`updateBookingAction` checks for changed `calendarTriggerFields` and calls `updateCalendarEvent()` if `calendar_event_id` exists. The event's date range updates correctly. But the event description includes call time and wrap time — if only dates change (call/wrap unchanged), the description stays correct. However if dates AND times change in the same edit, it's fine too. This is not a bug — noting for completeness.

### P2 | Compliance dashboard revalidation not confirmed

When talent completes onboarding, `submitOnboardingAction` (in `lib/data/onboarding.ts`) sets `onboarding_completed = true` but does not call `revalidatePath('/settings/compliance')`. The compliance page may show the old "incomplete" status until the Next.js cache expires naturally.

**Fix:** Add `revalidatePath('/settings/compliance')` to `submitOnboardingAction`.

### Three biggest gaps vs HoneyBook/Studio Ninja/Ignition

1. **Invoice delivery** — HoneyBook sends invoices with one click (integrated email + Stripe). Atelier requires: PDF download → Gmail → attach → send manually. Zero audit trail, zero automation.

3. **Client contact log** — Ignition and HoneyBook both have a timestamped activity log on every client (calls logged, emails sent, meetings noted). Atelier has only a free-text notes field. Institutional memory about client conversations lives in email/WhatsApp.

### Biggest architectural risk at 10× volume

The approval queue is a bottleneck. At 6 bookings/week with 5 crew each, that's 30 individual hold-request approvals per week that require Jasper to open the inbox, read each item, and click approve. At 60 bookings/week, that's 300 approvals for holds alone — before counting quote chase, compliance pings, and gallery shares.

The solution is already in the doctrine: preferred-core crew with unchanged templates and >48h lead time should auto-approve with kill-switch override available. This hasn't been implemented — the doctrine describes it but the code doesn't enforce it. This is the single highest-leverage change for Jasper's weekly time.

### Single change that saves the most time per week

**Reconnect Gmail + add failure visibility banner.** Based on the audit log, at least 5 emails have failed in the last 8 days — clients not getting quotes, talent not getting onboarding links, chase emails not sending. The automation system is running but its output is silently going nowhere. All the cron work and approval workflow is wasted if Gmail is down. This is the first fix, before anything else.
