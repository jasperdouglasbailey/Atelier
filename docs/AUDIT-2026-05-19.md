## 3. Currency precision

**What the code does today:**
- All money is `number` (JavaScript double-precision float).
- `formatCurrency(amount: number)` is display-only (Intl.NumberFormat).
- The fee engine wraps every operation in `r2(x) = Math.round(x * 100) / 100`
  which rounds to cents after each step.

**Where this can bite:**
- `0.1 + 0.2 === 0.30000000000000004` — classic float error. With r2,
  one operation rounds cleanly. But: chains of operations + sums of
  many rounded values can drift by a cent or two.
- Compounding: 20% commission × 10% GST × cost. If cost is stored as
  a value whose binary representation isn't exact, the round trip
  introduces error.

**Where this WON'T bite at Saunders & Co's scale:**
- Real shoots run 5–15 fee lines, sums in the $5k–$30k range, max
  decimals=2. The r2 rounding catches the obvious cases.
- Reconciliation check on the quote breakdown (Grand Total - Agency
  keeps - Owed to ATO - Paid through = $0.00) currently passes on the
  Testino booking. If float drift were a real issue, this check would
  show ±$0.01 mismatches.

**Industry-standard fix:** store currency as integer cents (so $1,234.56
becomes 123456 bigint). Adopt at a single chokepoint (FeeLine.subtotal
is bigint internally; the API hands out dollars via a converter).

**Recommendation:** not urgent. Defer until Saunders & Co's annual
revenue or per-booking value crosses ~$1M (at which point a $0.05
rounding error per quote becomes worth fixing). Document the risk in
CLAUDE.md so future-you knows to revisit.

**Severity:** P2

## 4. GDPR / right-to-be-forgotten — REAL GAPS FOUND

**What exists** (`src/app/actions/entities.ts`):
- `anonymiseTalentAction` clears ~25 PII fields and trashes the Drive
  folder.
- `anonymiseClientAction` clears name/email/phone/abn/notes + Drive.
- `anonymiseCrewAction` clears ~15 PII fields + Drive.
- All three audit-log `reason: 'right_to_be_forgotten'`.

**Gaps in `anonymiseTalentAction`** (PII columns left intact):
- **`nicknames[]`** — informal names that identify the artist. Used by
  the brief-detection heuristic. **PII per APP 1.**
- **`bank_account_name`** — full name attached to the bank account.
  Financial PII. **APP 12 / FDC-PoA implications.**
- **`bank_bsb`** — six-digit sort code, identifies the bank branch.
  Financial PII.
- **`bank_account_number`** — the account number itself. Financial PII.
- **`representation_status`** — context-PII ("represented by X agency",
  "freelance", etc.). Less critical.
- **`default_day_rate`, `min_day_rate`, `max_day_rate`** — financial
  history specific to the person. Arguable — could be left as
  "anonymised rate stats" or cleared. Lean toward clearing.

**Gaps in `anonymiseCrewAction`**: same shape — `default_day_rate` and
nicknames-equivalent fields, plus `bank_*` if present on the crew table
(need to verify the Crew interface).

**Severity:** P1 — if you ever process a deletion request from an artist
or crew member and a regulator asks "did you anonymise their bank
details?", the answer right now is no.

**Recommendation:** one focused PR adding the missing columns to the
update payloads + a unit test that compares the anonymise payload's
key set against the full set of PII columns on the Talent interface
(so future column additions can't sneak past silently).

## 5. Tax-year boundaries on reports

**What the code does** (`src/lib/data/reports.ts`):
- `revenue_this_year` filters by `new Date(r.created_at).getFullYear() === thisYear`.
- "This year" = **calendar year (Jan-Dec)**.

**What it should do for AU compliance:**
- Australian fiscal year is **1 July to 30 June**.
- A booking on 5 July 2026 belongs to FY2027 (the year ending 30 June 2027).
- Treasury reports, BAS, EOFY summaries are all FY-based.

**Real consequence:**
- The Reports page right now misleads on FY revenue. Anything in Jan-Jun
  is counted in the wrong fiscal year for tax planning.
- BAS / EOFY workflows: you'd need to hand-recalculate from Xero.

**Recommendation:** add a `fyOf(date)` helper (`if month >= 7 then year+1 else year`).
Add a `revenue_this_fy` field alongside `revenue_this_year`. Add a
calendar-year vs financial-year toggle to the Reports page. ~1 hour.

**Severity:** P1 (only because EOFY is real money)

## 6. Performance budget — bundle size

**What I checked:**
- `node_modules` weighs 640 MB on disk (does NOT ship — devDeps + node
  binaries).
- Heavy runtime deps from `package.json`:
  - `@sparticuz/chromium` — used for puppeteer PDF rendering. Hopefully
    only loaded on the PDF route, not in the main bundle.
  - `puppeteer-core` — same.
  - `recharts` — Reports page only.
  - `react-leaflet` + `leaflet` — Locations map page only.
  - `web-push` — VAPID for push notifications (server-only).

**What I didn't check** (would need a real `next build` run):
- The actual JS bundle size per route.
- First Load JS for `/bookings/[id]` — the QuoteBuilder is a massive
  client component, this will be the heaviest page.
- Whether code splitting is actually applied to recharts / react-leaflet.

**Recommendation:**
1. Run `npm run build` locally; capture the per-route "First Load JS" table
   that Next.js prints. Anything over 200 KB is a smell.
2. Enable Vercel Analytics + Speed Insights for real-user numbers
   (free tier exists).
3. If `/bookings/[id]` is >300 KB First Load, split QuoteBuilder out
   into a dynamic import or a route-segment with its own loading.tsx.

**Severity:** P2 (current users are on fast desktop; will bite if/when
multi-tenant SaaS launches)
